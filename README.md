## 1.1 你熟悉的 AI vs Claude Code

> 💡 **一句话定位**：Claude Code 不是代码补全工具，也不是 AI 聊天机器人——它是一个**运行在终端里的项目级 AI 工程助手**。
> 
> 它能读懂你整个项目、自己做计划、动手改代码、执行命令、检查结果，直到把活干完。

2022 年底，ChatGPT 横空出世。

三年多过去了，我们每个人对这种 AI 的用法都很熟悉了——你问一句，它答一句。你觉得答得不好，换个问法再问一次。

这种「一问一答」的模式，帮你查资料、写文案、翻译文字都没问题。但遇到真正需要「干活」的事情，它就无能为力了。

这就是 **AI 聊天机器人** 和 **AI 智能体（Agent）** 的本质区别。

**💡「简单来说」：** 聊天 AI = 只动嘴的军师，Agent = 能自己动手干活的助手。

## 1.2 Agent 的秘密：大模型循环

Claude Code 为什么能做到「自己干活」？我们来拆解它的核心机制。

### 大脑 + 手脚 = Agent

要理解 Claude Code，你先认识三个角色：

| 角色 | 是什么 | 能干嘛 | 举例 |
|------|--------|--------|------|
| **大模型（LLM）** | AI 的大脑 + 眼睛 | 理解你的需求、观察环境、做出决策 | "我应该先创建一个 HTML 文件" |
| **工具（Tools）** | AI 的手脚 | 读写文件、搜索网络、执行命令、调用服务 | 真的在你的电脑上创建一个文件 |
| **Agent 程序** | 连接大脑和手脚的神经系统 | 把大模型的决策翻译成工具调用，再把结果反馈给大模型 | Claude Code 本身 |

🔰「新手建议」：你不用记住所有工具的名字——现在只需要理解这个「大脑指挥手脚干活」的比喻就够了。

### LLM Loop：AI 的工作循环

普通聊天 AI 是这样工作的：

```
你提问 → AI 回答 → 结束
```

Claude Code 是这样工作的：


![LLM Loop 循环示意图：思考 → 选工具 → 执行 → 反馈 → 再思考，形成闭环](第一章/assets/llm-loop-diagram.png)

这个过程就叫 **LLM Loop（大模型循环）**。它让 AI 从「一问一答」变成了「持续干活」。

### 为什么 Claude Code 特别聪明？

市面上有很多 Agent 产品，但 Claude Code 的循环设计得特别出色。用一个比喻：

- **普通 Agent** = 实习生。你推他一步，他走一步。推得不明确，他就停了。
- **Claude Code** = 成熟的职场人。你给他一个目标，他自己规划步骤、调用工具、执行操作、检查结果。卡住了？他会自己上网搜索找解决方案。遇到报错？他会自己 Debug。

![实习生 Agent vs Claude Code：左边需要每一步指令，右边自主完成全流程](第一章/assets/intern-vs-pro-agent.png)

这就是「自主决策循环」：**感知环境 → 推理决策 → 调用工具 → 评估结果 → 继续循环**。整个过程你只需要在开始说一句话，中间最多点几次「同意」。

## 1.3 Claude Code 和别的工具有什么不同

市面上 AI 编程工具不少，我们来对比一下，帮你快速建立坐标系。

### **Claude Code 与 Copilot 的区别**

很多人会问：既然 Copilot 也能帮我写代码，为什么还要用 Claude Code？

核心区别在于：Copilot 更像一个**贴身助理**——你写代码时它在旁边看着，猜你接下来要写什么，帮你补全。但 Claude Code 更像一个**外包团队**——你把整个需求丢给它，它自己规划、写代码、跑测试、修 Bug，最后交给你一个能用的东西。

Copilot 适合帮你**写得快**。

Claude Code 适合帮你**做完事**。

更合理的用法是：日常编码用 Copilot 提效，复杂项目或从零开发时用 Claude Code 完整交付。
![Claude Code vs Copilot](第一章/assets/cc-vs-copilot.png)

### **Claude Code 与 Cursor 的区别**

很多人会问：Cursor 不也能在编辑器里跟 AI 对话改代码吗，CC 有什么不同？

核心区别在于：Cursor 更像一个**会 AI 的编辑器**——你在编辑器里写代码，AI 帮你改、帮你补、帮你解释。但 Claude Code 更像一个**会用编辑器的 AI**——它不依赖你手动打开文件、选中代码，你只需要说目标，它自己决定要改哪些文件、执行什么命令。

Cursor 适合**你在写代码时辅助你**。

Claude Code 适合**你不用写代码时替代你**。

更合理的用法是：用 Cursor 做轻量编辑和代码审查，用 Claude Code 做重度开发和完整项目交付。![Claude Code vs Cursor](第一章/assets/cc-vs-cursor.png)

### **Claude Code 与 Codex 的区别**

很多人会问：Codex 也是命令行 Agent，也能自主干活，和 CC 有什么不同？

核心区别在于：Codex 更像 OpenAI 生态里的**工程助手**——它背后是 GPT 系列模型，深度绑定 OpenAI 的工具链。而 Claude Code 更像 Anthropic 生态里的**长期工程伙伴**——它不仅有 Claude 模型驱动，还有更成熟的 Harness 设计、更丰富的 Skills/Hooks/Plugins 生态，以及更灵活的模型接入（支持第三方）。

Codex 适合**已经在 OpenAI 生态中的开发者**。

Claude Code 适合**想要最成熟 Agent 体验、不想被单一模型绑定的用户**。

更合理的用法是：如果你已经深度使用 OpenAI 全家桶，Codex 是不错的延伸；如果你想体验当前最强的 Agent 驾驭系统，Claude Code 是更好的选择。


![Claude Code vs Codex：Anthropic 生态 vs OpenAI 生态](第一章/assets/cc-vs-codex.png)

| 工具                     | 一句话概括               | 收费      | 强项                  | 弱项            |
| ---------------------- | ------------------- | ------- | ------------------- | ------------- |
| **GitHub Copilot**     | 代码自动补全插件            | 付费      | 在你写代码时智能预测下一行       | 只能补全，不能自主干活   |
| **Cursor**             | AI 增强版 VS Code      | 付费      | 在编辑器内对话 + 改代码，体验流畅  | 付费额度有限制       |
| **Codex / OpenAI CLI** | OpenAI 的命令行 Agent   | 付 API 费 | 背后是 GPT 系列模型        | 使用门槛较高        |
| **Claude Code**        | Anthropic 的终端 Agent | 订阅或 API | 自主干活能力最强，harness 最好 | 终端界面，小白上手需要适应 |

💡「简单来说」：Copilot 帮你写一段代码，Cursor 帮你改一个文件，Claude Code 帮你做完整个项目。

🔰「新手建议」：不用纠结「我该用哪个」——这本书就是教你用好 Claude Code 的。掌握它之后，其他工具你也能很快上手。

## 1.4 为什么是「Harness 的集大成者」

你可能听过一个词——**Harness（驾驭系统）**。

用大白话解释：Harness 就是 Agent 这个「身体」的设计水平。同样的大脑（大模型），装在不同的身体里，表现天差地别。

![同样的模型、不同的身体：ChatGPT网页版 → 普通Agent → Claude Code，能力逐级递增](第一章/assets/same-brain-different-bodies.png)

举个例子：同样是调用 Claude Sonnet 模型：

- 你在 **Claude 网页版**里问它 → 它只能给你文字建议
- 你通过 **Claude Code** 调用同一个模型 → 它能读你的整个项目、修改文件、运行命令、测试结果

一样的脑子，不一样的身体，能干的事完全不一样。

Claude Code 的 Harness 设计好在哪？几个关键点：

- 🔄 **自主循环**：不是推一步走一步，而是自己持续推进直到任务完成
- 📂 **终端原生**：直接运行在你的电脑上，不受浏览器沙箱限制
- 🧠 **超长上下文**：能记住整个项目的来龙去脉，不会做着做着「失忆」
- 🛡️ **权限清晰**：什么操作要问你、什么可以直接做，边界分明
- 🧩 **官方 Skill**：Anthropic 团队提前设计了大量开箱即用的技能包

🔰「新手建议」：不用深究 Harness 这个词。你只需要知道——Claude Code 不只是给大模型套了个壳，它的「身体」经过精心设计，这才是它真正厉害的地方。

## 1.5 Claude Code 能做什么

![三类人三种用法：完全不会编程的人 → 会一点编程的人 → 专业开发者](第一章/assets/three-personas.png)

知道了 Claude Code 的原理，你可能会想：它具体能帮我干什么？

使用 Claude Code 的人大致分三类，每一类用它做完全不同的事。

**第一类：完全不会编程的人**

这群人用 CC，不是为了写代码，是为了**省掉重复劳动**。对他们来说，CC 就是一个「能操作电脑的 AI」。

他们用它做的事包括：

- 整理文件夹：批量重命名、分类归档、删除重复文件
- 处理表格：合并 Excel、筛选数据、生成图表
- 写东西：公众号文章、周报、邮件、PPT 大纲
- 搜资料：总结网页内容、对比多篇文章、提取关键信息
- 设置提醒：定时任务、自动检查、格式校验

🔰「新手建议」：如果你是这一类，不用学任何编程知识。你只需要学会两件事——怎么把文件放到 CC 能看到的文件夹里，以及怎么用自然语言说清楚你要什么。

---

**第二类：会一点编程，但不想什么都自己写**

这群人可能是学生、设计师、产品经理，或者自学过一点前端。他们能用 CC **加速自己本来就做得到的事**，做那些「会做但懒得做」的工作。

他们用 CC 的典型场景：

- 做一个网页工具（番茄钟、倒计时、表单收集）
- 写一个数据爬虫脚本
- 搭一个博客或作品集网站
- 把设计稿截图丢给 CC，让它生成前端页面
- 给已有的项目加一个小功能

💡「简单来说」：你会指挥，CC 会执行。你不需要记住每一个 CSS 属性，只需要能判断结果对不对。

---

**第三类：专业开发者**

对程序员来说，CC 的角色更像是**一个能独立干活的队友**。它不只是代码补全，而是：读项目 → 理解架构 → 出方案 → 写代码 → 跑测试 → 看结果 → 修问题，自己完成一整圈。

专业开发者常用 CC 做：

- 接手新项目时的快速上手（让 CC 读一遍项目，生成架构说明）
- 修 Bug（描述现象 + 贴报错日志，CC 自己定位并修复）
- 写测试（把测试覆盖率从 40% 补到 80%）
- 代码审查（让 CC 检查 diff，指出潜在问题）
- 重构小模块（限定范围，要求不改行为只改结构）
- 写文档和 PR 描述（自动生成 commit message 和变更说明）

⚠️「注意」：程序员用 CC，关键不是让它替你思考，而是让它替你**执行**。你控制方向和边界，它负责落地细节。

---

### 不只是编程——CC 的非代码能力

名字里带「Code」让很多人误以为它只能写代码。实际上越来越多人在用它做跟代码毫无关系的事。

我们调研了多个 Claude Code 深度用户，以下是他们最常提到的非编程场景：

**内容创作**

公众号写作者用 CC 做选题调研：把几篇参考文章丢进去，让 CC 提炼核心观点、找出差异化角度、生成大纲。视频创作者用 CC 分析脚本结构、匹配画面素材、统计时长节奏。

**数据整理**

不需要打开 Excel 写公式。把 CSV 或 Excel 文件放到项目文件夹，用自然语言告诉 CC 你想怎么处理。它会自己选工具（Python 脚本、命令行工具、甚至直接读写），你只需要看结果。

**自动化工作流**

重复超过三次的事情，就可以让 CC 帮你自动化。比如：每次发布文章前自动检查标题字数、封面尺寸、标签格式；每天早上自动汇总昨天的业务数据发到飞书群。

💡「简单来说」：**只要你的工作流程是「读东西 → 想一下 → 写东西 → 发出去」，CC 就能介入其中任何一个环节。**

---

### 什么时候不该用 Claude Code

说清楚能力之后，也要说清楚边界。Claude Code 不是万能的，以下场景**不要**用它：

- 🚫 **操作生产数据库**：一行错误的 SQL 足够让你加班到天亮
- 🚫 **处理支付和资金**：任何涉及钱的逻辑，AI 的幻觉概率不可接受
- 🚫 **动安全模块**：认证、加密、权限这些，需要人类审计
- 🚫 **你无法验收的任务**：如果你判断不了它做得对不对，就别让它独立做

🔰「新手建议」：一句话记住——**CC 是你的队友，不是你的替身。你永远要对最终结果负责。**

---

### 一句话总结

Claude Code 能做的不只是写代码。它真正的价值在于：**让一个不会编程的人能操作电脑完成任务，让一个会编程的人能把精力从执行层提升到决策层。**

你不必成为程序员才能用它，也不必害怕终端界面。它归根到底就是一个工具——一个你说人话、它帮你干活的工具。

## 1.6 运行在本地，这件事很重要

![大脑在云端身体在本地：网页版 AI vs Claude Code 的架构对比](第一章/assets/local-vs-cloud.png)

还有一个被很多人忽视的关键区别：Claude Code 是**运行在你的电脑上**的程序，而不是一个网页服务。

这意味着什么？

| | 网页版 AI | Claude Code |
|---|---|---|
| **操作文件** | 你需要手动上传、下载 | 直接读写你电脑里的文件 |
| **执行命令** | 不能 | 可以用你的终端做任何事 |
| **权限控制** | 无（你自己负责） | 四种模式，精细控制 |
| **隐私** | 对话内容上传到云端 | 大脑在云端推理，但身体在本地执行 |

💡「简单来说」：CC 的大脑在云端（调用 API 推理），但身体在你的电脑上（操作真实文件）。这个分工，让它比任何纯网页 AI 产品都强大。

> **本章定位**：安装之前先做选择。Claude Code 有两种完全不同的使用方式——命令行（CLI）和桌面应用（Desktop）。本章帮你搞清楚两者的区别、各自适合谁、能不能一起用。

> **预计阅读时间**：10 分钟

### 2.1 两种模式，同一颗大脑

很多人以为「Claude Code Desktop」就是给终端版套了个图形壳——这个理解是错的。

实际上，Claude Code 有两种独立的「身体」：

- **CLI（命令行界面）**：运行在终端里，纯文本交互，通过 `claude` 命令启动
- **Desktop APP（桌面应用）**：独立的 Mac/Windows 应用程序，带完整图形界面

它们共用同一颗「大脑」——背后都是同一个 Claude Code 引擎，调用的都是 Claude 系列模型。但两种身体的构造完全不同，适合的场景也完全不同。

![CLI vs Desktop 总览对比 — 一颗大脑，两种身体](第二章/assets/一颗大脑两种身体.png)

![命令行 CLI vs 桌面版APP](第二章/assets/命令行 CLI vs 桌面版APP.png)

在决定装哪个之前，先看一张总表建立概念：

| 项目            | 🖥️ CLI（命令行）                      | 🖼️ Desktop APP（桌面应用）                     |
| ------------- | --------------------------------- | ----------------------------------------- |
| **界面**        | 终端 TUI / 文本交互                     | GUI：多窗格（chat / diff / terminal / preview） |
| **核心能力**      | 完整能力（文件操作 / git / 工具调用）           | 完整能力（同引擎）                                 |
| **安装方式**      | npm / brew / CLI 安装               | dmg / exe 安装包                             |
| **文件/代码操作**   | 自动读写 repo，可直接运行 shell 命令          | 可视化文件操作 + 内置编辑器                           |
| **图片/PDF/附件** | 支持但偏"路径/命令式"                      | ✔ 直接拖拽 / 粘贴（体验更好）                         |
| **Diff 查看**   | git diff / 文本 diff                | ✔ 可视化逐行 diff（更强）                          |
| **多会话管理**     | 手动 terminal tabs / tmux           | ✔ 自动并行 session + worktree 隔离              |
| **终端能力**      | ✔ 原生 shell / CI / cron / pipeline | ✔ 内置 terminal（但偏交互，不用于 CI）                |
| **自动化脚本**     | ✔ 强（CI/CD / cron / API workflow）  | ❌ 不适合自动化                                  |
| **应用预览**      | 外部浏览器                             | ✔ 内置 preview（HTML/PDF/前端）                  |
| **模型选择**      | CLI 参数 / config                   | UI 下拉选择                                   |
| **配置共享**      | ~/.claude.json / MCP / CLAUDE.md  | ✔ 与 CLI 共享配置                              |
| **权限控制**      | 更细（可用于自动批准等）                      | GUI 模式控制（更安全但更限制）                         |
| **Linux 支持**  | ✔ 支持                              | ❌ 不支持                                     |
| **适合人群**      | 工程化 / 自动化 / 重度开发                  | 交互式开发 / 多任务 / 代码审查                        |

💡「简单来说」：CLI 功能最强但需要动手能力，Desktop 最好上手但相对没有那么灵活。

### 2.2 CLI 模式详解 — 终端里的 AI 工作台

CLI 是 Claude Code 最原生的形态。启动方式就是打开终端，输入：

```bash
claude
```

然后你会看到一个纯文本的对话界面。没有侧边栏、没有文件树、没有按钮——只有文字。但这个看似简陋的界面，是所有模式里**功能最完整的**。

![CLI 三种工作模式 — 交互对话 · 脚本管道 · 定时自动化](第二章/assets/CLI三种打开方式.png)

#### CLI 的独特优势

**① 脚本化 — 一行命令干完活**

CLI 最强大的能力不在对话里，而在 `--print`（简称 `-p`）模式。你可以用一行命令让 CC 直接执行任务并输出结果，不需要进入交互界面：

```bash
# 分析日志文件
cat error.log | claude -p "分析这段日志，找出所有 Error 的根因"

# 一键生成 commit message
git diff | claude -p "根据这个 diff 写一句中文 commit message"

# 批量处理文件
ls *.csv | claude -p "读取这些 CSV 文件，合并后生成汇总表"
```

📋「提示词模板」：`claude -p "你对一段文字的任务描述"` — 可以接在任何管道后面。

**② 自动化与 CI/CD**

CLI 可以编入脚本、cron 定时任务、GitHub Actions 流水线。比如：

```bash
# 每天早上 9 点自动跑测试并生成报告
0 9 * * * cd /path/to/project && claude -p "运行所有测试，如果失败发飞书通知"
```

Desktop 不能做这些——它只能你手动打开、手动操作。

**③ 免费起步，Linux 可用**

CLI 免费账户就能用，只是有使用额度限制。而且是唯一支持 Linux 的方式——Desktop 只提供 macOS 和 Windows 版。

#### 模型选择：CLI 和 Desktop 都支持第三方

值得一提的是，**两种模式都可以接入第三方模型**。CLI 配合 CC Switch、Desktop 在设置中配置后在下拉菜单切换——DeepSeek、豆包、Agnai 等国产和免费模型都能用。本书安装章节会详细讲解配置方法。

#### CLI 的局限

- **不能直接粘贴图片**：需要先把截图存为文件，再用文件路径引用
- **纯文本 Diff**：代码改动只能看文字版 + - 对比，没有可视化高亮
- **对新手不够友好**：终端界面容易被误解为「程序员专属」

🔰「新手建议」：终端本质就是一个「跟电脑说话的白板」，不是写代码专用。你只需要会打字和复制粘贴就够了。

### 2.3 Desktop 桌面版详解 — 独立 App 体验

Claude Code Desktop 发布于 2025 年下半年，并在 2026 年 4 月经历了一次重大改版。

现在的 Desktop 是一个完整的桌面应用程序，打开它就像打开 VS Code 或任何一款 Mac 软件。

#### 为什么桌面版 APP 对新手更友好

如果你跟我一样，从小到大用的都是「可视化应用」——微信是图标、Word 是图标、浏览器是图标——那么 Desktop 就是你的舒适区。我们这一代人习惯了「看到什么点什么」，不习惯「对着黑窗口打字」。

几个真实场景你就明白差别有多大：

**打开一个项目：**

🖼️ Desktop：点击菜单栏 → 文件 → 打开文件夹 → 在 Finder 里选目标文件夹 → 搞定。
🖥️ CLI：打开终端 → 输入 `cd /Users/你的名字/Documents/项目名` → 路径打错了 → 重来。

**拖一个文件进去让 AI 处理：**

🖼️ Desktop：从 Finder 把 PDF 拖进对话框 → AI 自动读取。
🖥️ CLI：记住文件路径 → 手动输入 `@/Users/你的名字/Downloads/报告.pdf` → 确认路径没错。

**贴一张截图让 AI 改 UI：**

🖼️ Desktop：截图 → Cmd+V 粘贴 → 「帮我把这个按钮往右移 10px」。
🖥️ CLI：截图 → 保存到桌面 → 记住文件名 → 输入文件路径 → 告诉 AI 去读。

你会发现，Desktop 的操作逻辑跟微信发图、Word 拖文件一模一样。零学习成本。

🔰「新手建议」：如果你以前看到「终端」两个字就下意识回避，**先从 Desktop 开始**。等你在 App 里熟悉了 CC 的工作方式，再去学 CLI 就只是换了个界面，不会觉得困难。Desktop 是你理解 Agent 工作方式的最佳过渡。

> 💡 一句话：Desktop 让你把精力花在「想清楚要做什么」上，而不是「怎么操作这个工具」。

![Desktop 操作体验 — 拖进去 · 贴上去 · 点出来](第二章/assets/Desktop拖进去贴上去点出来.png)

#### Desktop 的局限

- **不能自动化**：Desktop 是**交互式 GUI 工具，不支持 CLI 的 headless/脚本模式（如 `-p`）**
- **仅支持 macOS 和 Windows**：Linux 用户只能用 CLI

### 2.4 可以同时用吗？当然可以

很多人会同时使用 CLI 和 Desktop，在不同场景切换，两者属于**同一套 Claude Code 体系的不同交互方式**。

但关键点是：

> ✔ 它们确实"部分共享配置"  
> ❌ 但不是"全部共享记忆与状态"

#### 哪些是共享的？

在 Claude Code 体系中，以下内容是**CLI 与 Desktop 共用或自动同步生效的：**

**项目级与用户级配置**

- `CLAUDE.md`（项目上下文与规则）
- `CLAUDE.local.md`（本地补充规则）
- `settings.json`（权限、行为、模型等设置）
- `.claude/settings.json`（项目级设置）

这些属于同一配置体系，会在不同客户端中生效

**扩展能力与行为配置**

- Hooks（事件触发脚本）
- Skills（可复用能力模块）
- Sub-agents（子代理）

这些机制在 CLI 与 Desktop 中共享行为定义
