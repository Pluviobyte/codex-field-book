# CLI 还是 Desktop？选择你的使用模式

> **本章定位**：安装之前先做选择。Claude Code 有两种完全不同的使用方式——命令行（CLI）和桌面应用（Desktop）。本章帮你搞清楚两者的区别、各自适合谁、能不能一起用。
>
> **预计阅读时间**：10 分钟

---

## 2.1 两种模式，同一颗大脑

很多人以为「Claude Code Desktop」就是给终端版套了个图形壳——这个理解是错的。

实际上，Claude Code 有两种独立的「身体」：

- **CLI（命令行界面）**：运行在终端里，纯文本交互，通过 `claude` 命令启动
- **Desktop APP（桌面应用）**：独立的 Mac/Windows 应用程序，带完整图形界面

它们共用同一颗「大脑」——背后都是同一个 Claude Code 引擎，调用的都是 Claude 系列模型。但两种身体的构造完全不同，适合的场景也完全不同。

![cli与桌面版APP对比](assets/CLIvs桌面版APP.png)

在决定装哪个之前，先看一张总表建立概念：

| 项目            | 🖥️ CLI（命令行）                      | 🖼️ Desktop APP（桌面应用）                     |
| ------------- | --------------------------------- | ----------------------------------------- |
| **界面**        | 终端 TUI / 文本交互                     | GUI：多窗格（chat / diff / terminal / preview） |
| **核心能力**      | 完整能力（文件操作 / git / 工具调用）           | 完整能力（同引擎）                                 |
| **安装方式**      | npm / brew / CLI 安装               | dmg / exe 安装包                             |
| **文件/代码操作**   | 自动读写 repo，可直接运行 shell 命令          | 可视化文件操作 + 内置编辑器                           |
| **图片/PDF/附件** | 支持但偏「路径/命令式」                    | ✔ 直接拖拽 / 粘贴（体验更好）                         |
| **Diff 查看**   | git diff / 文本 diff                | ✔ 可视化逐行 diff（更强）                          |
| **多会话管理**     | 手动 terminal tabs / tmux           | ✔ 自动并行 session + worktree 隔离              |
| **终端能力**      | ✔ 原生 shell / CI / cron / pipeline | ✔ 内置 terminal（但偏交互，不用于 CI）                |
| **自动化脚本**     | ✔ 强（CI/CD / cron / API workflow）  | ❌ 不适合自动化                                  |
| **应用预览**      | 外部浏览器                             | ✔ 内置 preview（HTML/PDF/前端）                 |
| **模型选择**      | CLI 参数 / config                   | UI 下拉选择                                   |
| **配置共享**      | ~/.claude.json / MCP / CLAUDE.md  | ✔ 与 CLI 共享配置                              |
| **权限控制**      | 更细（可用于自动批准等）                      | GUI 模式控制（更安全但更限制）                         |
| **Linux 支持**  | ✔ 支持                              | ❌ 不支持                                     |
| **适合人群**      | 工程化 / 自动化 / 重度开发                  | 交互式开发 / 多任务 / 代码审查                        |

![CLI vs Desktop 总览对比 — 一颗大脑，两种身体](assets/一颗大脑两种身体.png)

💡「简单来说」：CLI 功能最强但需要动手能力，Desktop 最容易上手但相对没有那么灵活。

## 2.2 CLI 模式详解 — 终端里的 AI 工作台

CLI 是 Claude Code 最原生的形态。启动方式就是打开终端，输入：

```bash
claude
```

然后你会看到一个纯文本的对话界面。没有侧边栏、没有文件树、没有按钮——只有文字。但这个看似简陋的界面，是所有模式里**功能最完整的**。

![CLI 三种工作模式 — 交互对话 · 脚本管道 · 定时自动化](assets/CLI三种打开方式.png)

### CLI 的独特优势

**① 脚本化 — 一行命令干完活**

CLI 最强大的能力不在对话里，而在 `--print`（简称 `-p`）模式。你可以用一行命令让 CC 直接执行任务并输出结果，不需要进入交互界面：

```bash
# 分析日志文件
cat error.log | claude -p "分析这段日志，找出所有 Error 的根因"

# 一键生成 commit message
git diff | claude -p "根据这个 diff 写一句中文 commit message"

# 批量处理文件
ls *.csv | claude -p "读取这些 CSV 文件，合并后生成汇总表"
```

📋「提示词模板」：`claude -p "你对一段文字的任务描述"` — 可以接在任何管道后面。

**② 自动化与 CI/CD**

CLI 可以编入脚本、cron 定时任务、GitHub Actions 流水线。比如：

```bash
# 每天早上 9 点自动跑测试并生成报告
0 9 * * * cd /path/to/project && claude -p "运行所有测试，如果失败发飞书通知"
```

Desktop 不能做这些——它只能你手动打开、手动操作。

**③ 免费起步，Linux 可用**

CLI 免费账户就能用，只是有使用额度限制。而且是唯一支持 Linux 的方式——Desktop 只提供 macOS 和 Windows 版。

### 模型选择：CLI 和 Desktop 都支持第三方

值得一提的是，**两种模式都可以接入第三方模型**。CLI 配合 CC Switch、Desktop 在设置中配置后在下拉菜单切换——DeepSeek、豆包、智谱 GLM 等国产和免费模型都能用。本书安装章节会详细讲解配置方法。

### CLI 的局限

- **图片处理不如 Desktop 顺手**：虽然新版 CLI 已支持粘贴图片（macOS: Ctrl+V / Windows: Alt+V），但相比 Desktop 拖拽 + 直接粘贴的体验仍显繁琐
- **纯文本 Diff**：代码改动只能看文字版 + - 对比，没有可视化高亮
- **对新手不够友好**：终端界面容易被误解为「程序员专属」

🔰「新手建议」：终端本质就是一个「跟电脑说话的白板」，不是写代码专用。你只需要会打字和复制粘贴就够了。

## 2.3 Desktop 桌面版详解 — 独立 App 体验

Claude Code Desktop 发布于 2025 年下半年，并在 2026 年 4 月经历了一次重大改版。

现在的 Desktop 是一个完整的桌面应用程序，打开它就像打开 VS Code 或任何一款 Mac 软件。

### 为什么桌面版 APP 对新手更友好

如果你跟我一样，从小到大用的都是「可视化应用」——微信是图标、Word 是图标、浏览器是图标——那么 Desktop 就是你的舒适区。我们这一代人习惯了「看到什么点什么」，不习惯「对着黑窗口打字」。

几个真实场景你就明白差别有多大：

**打开一个项目：**

🖼️ Desktop：点击菜单栏 → 文件 → 打开文件夹 → 在 Finder 里选目标文件夹 → 搞定。
🖥️ CLI：打开终端 → 输入 `cd /Users/你的名字/Documents/项目名` → 路径打错了 → 重来。

**拖一个文件进去让 AI 处理：**

🖼️ Desktop：从 Finder 把 PDF 拖进对话框 → AI 自动读取。
🖥️ CLI：记住文件路径 → 手动输入 `@/Users/你的名字/Downloads/报告.pdf` → 确认路径没错。

**贴一张截图让 AI 改 UI：**

🖼️ Desktop：截图 → Cmd+V 粘贴 → 「帮我把这个按钮往右移 10px」。
🖥️ CLI：截图 → 保存到桌面 → 记住文件名 → 输入文件路径 → 告诉 AI 去读。

你会发现，Desktop 的操作逻辑跟微信发图、Word 拖文件一模一样。零学习成本。

🔰「新手建议」：如果你以前看到「终端」两个字就下意识回避，**先从 Desktop 开始**。等你在 App 里熟悉了 CC 的工作方式，再去学 CLI 就只是换了个界面，不会觉得困难。Desktop 是你理解 Agent 工作方式的最佳过渡。

💡「简单来说」：Desktop 让你把精力花在「想清楚要做什么」上，而不是「怎么操作这个工具」。

![Desktop 操作体验 — 拖进去 · 贴上去 · 点出来](assets/Desktop拖进去贴上去点出来.png)

### Desktop 的局限

- **不能自动化**：Desktop 是**交互式 GUI 工具，不支持 CLI 的 headless/脚本模式（如 `-p`）**
- **仅支持 macOS 和 Windows**：Linux 用户只能用 CLI

## 2.4 可以同时用吗？当然可以

短答：**完全可以**，而且推荐两个都装。装了 Desktop 之后 CLI 不用卸载——两者可以在同一个项目上并排工作。

先澄清一件容易混淆的事：**Claude Desktop 应用里其实有三个 tab**：

- **Chat** — 普通对话（就像你用 claude.ai）
- **Cowork** — Agent 模式，处理长任务
- **Code** — 这就是「Claude Code Desktop」的完整界面

本节说的「Desktop 与 CLI 共享」，指的是 **Code tab 与 CLI 之间的关系**。

![CLI + Desktop 共享配置 — 装两个，配一次](assets/装两个配一次.png)

### 共享的机制：读同一份文件

「共享」不是两个软件在互相同步，而是**两端都从相同的路径读取配置文件**。所以你在 CLI 装的 Skill、写的 `CLAUDE.md`，Desktop 一打开就直接生效，反过来也一样。

用官方文档的话：

> Desktop and CLI read the same configuration files, so your setup carries over.

### 真正共享的清单

| 类别               | 存放位置                                                   |
| ---------------- | ------------------------------------------------------ |
| 项目上下文            | `<项目>/CLAUDE.md`、`CLAUDE.local.md`                     |
| 项目级配置 / 权限       | `<项目>/.claude/settings.json`                           |
| 项目级 MCP          | `<项目>/.mcp.json`                                       |
| 用户级设置            | `~/.claude/settings.json`                              |
| 用户级 MCP          | `~/.claude.json`（的 `mcpServers` 段）                     |
| Skills           | `~/.claude/skills/`、`<项目>/.claude/skills/`             |
| Sub-agents（子代理） | `~/.claude/agents/`、`<项目>/.claude/agents/`             |
| Hooks            | 在 `settings.json` 里定义                                  |
| 插件               | `~/.claude/plugins/`                                   |

规律很简单：**所有 Claude Code 相关的能力扩展都在 `~/.claude/` 或项目内 `.claude/`，两端物理路径完全一致，装一次两边都能用。**

### 一个容易踩坑的例外

Desktop 应用还维护着一份**独立的 MCP 配置文件**：

```
~/Library/Application Support/Claude/claude_desktop_config.json
```

这份配置 Desktop 应用（Chat + Cowork + Code 三个 tab）都能读，**但 CLI 读不到**（CLI 只看 `~/.claude.json`）。所以如果你在 Desktop 的图形界面里点鼠标装了一个 MCP，想让 CLI 也用上，需要执行一次导入命令：

```bash
claude mcp add-from-claude-desktop
```

这条命令会把 `claude_desktop_config.json` 里的 MCP 服务器复制到 `~/.claude.json`，之后 CLI 就能看到了。

### 明确不共享的（官方原话）

> Each maintains separate session history.

具体来说：

- **会话历史**：CLI 的 `~/.claude/history.jsonl` 和 Desktop 的 session 数据各存各的，不会互相看到
- **当前进行中的任务与临时状态**：切换客户端等于从零开始一个新会话
- **窗口布局、pane 位置、UI 偏好**：Desktop 独有，与 CLI 无关

### 一个实用小技巧：把 CLI 会话一键搬到 Desktop

如果你正在 CLI 里跑一个任务，突然想改用图形界面看 diff、拖文件，直接在 CLI 里输入：

```bash
/desktop
```

当前 session 会自动保存并在 Desktop 里打开，CLI 自己退出。反过来 Desktop 没有对应的一键命令，但因为项目级配置本来就共享，重新在 CLI 里 `cd` 到同一个项目路径就能继续工作。

⚠️「注意」：**装两个、配一次，但对话记忆不会跨端接续。** 如果需要在两端之间传递对话产出，用 `CLAUDE.md`、Skills 或直接把结论写入项目文件——这样下次不管在哪端打开，都能读到。

## 本章要点

- ✅ Claude Code 有两种模式：**CLI（命令行）** 和 **Desktop（桌面应用）**，共用同一引擎
- ✅ CLI 自动化最强：脚本化、管道、CI/CD、Linux，免费可用
- ✅ Desktop 体验最好：可视化 Diff、并行会话、拖拽贴图、集成预览
- ✅ 第三方模型：CLI 和 Desktop 都支持 CC Switch 接入国产/免费模型
- ✅ 零基础用户推荐路径：先 Desktop 入门 → 后 CLI 进阶
- ✅ 两者可以同时安装，**配置自动共享**
